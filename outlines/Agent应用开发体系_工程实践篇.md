# Agent应用开发体系 - 工程实践篇

> 本篇聚焦 Agent 应用开发的实际工程流程，包括 AI 辅助开发工作流、测试方法论、CI/CD 自动化以及安全与合规实践。
>
> **不包含**：具体框架/工具用法（见工程篇）、模型训练/部署（见模型算法篇）、Agent 架构设计（见设计模式篇）

### 什么情况下优先归工程实践篇

1. 当知识点主要关注“怎么把 Agent 项目稳定地开发、测试、上线、治理”时，优先归工程实践篇。
2. 当重点是方法论、流程、规范、治理动作、上线运行经验，而不是底层技术构件本身时，优先归本篇。
3. 当同一个词既可以描述系统能力，也可以描述工程落地方式时，本篇优先收纳“落地方式”那一侧。
4. `outline` 中已有对应二级主题时，应先挂靠现有主题，再考虑是否补充三级锚点。

### 常见混淆项

| 容易混淆的概念 | 本篇归类 | 不归本篇的情况 | 判断关键点 |
|---|---|---|---|
| Harness Engineering vs Agent Harness | `Harness Engineering` 放 `project-management/` | `Agent Harness` 放工程篇 `core-concepts/` | 前者讲生产化工程体系，后者讲运行时控制层 |
| 安全防护实践 vs Guardrails | 风险控制、审批、注入防护、权限治理放 `security/` | SDK 自带 Guardrails 能力、运行时机制可放工程篇或 SDK 卡片 | 看重点是治理实践还是产品/框架能力 |
| Eval-driven Testing vs 评估框架/评测平台 | 测试流程、回归策略、测试设计放 `testing/` | RAGAS、DeepEval、LangSmith 这类评估平台/工具放生态工具篇 | 看重点是“测试方法”还是“评估工具” |
| Claude Code Hooks 的自动化使用 | 用于开发流程自动检查、PR 审查、规范执行可放本篇 | Hook 机制本身定义、事件模型归工程篇 | 看重点是流程治理还是机制介绍 |

### 最低锚点规则

- 二级标题代表 `content/04_practices/*` 的子目录主题。
- 三级标题代表可直接落卡的具体方法、流程、规范或实践专题。
- 没有同名三级条目时，只要已有合适二级主题，就应先补三级锚点而不是停住。

### 扩展主题锚点

如果新知识点无法直接归入现有三级条目，优先挂靠以下扩展容器：

- `二、Agent 测试方法`：承载新测试方法、评测驱动流程、验证策略
- `四、安全与合规`：承载生产治理、安全审查、审计与风控规则
- `六、项目管理实践`：承载运行规范、团队协作模式、生产化工程方法

### 粒度说明

- **二级标题 = 子目录主题**，用于确定目标落入哪个实践类目录。
- **三级标题 = 最小卡片单位**，可以是工作流、测试法、部署策略、治理机制或工程方法。
- 三级条目下的列表项是该卡片的最低覆盖面，不要求一张卡片写完全部列表，但必须围绕该主题展开。

### 6.4 Harness Engineering（Agent 生产化工程）
- 基本定义：围绕 Agent 构建可运行、可治理、可扩展生产系统的工程方法
- 与 Prompt Engineering、Context Engineering、Agent Harness 的区别
- 核心组成：运行时编排、工具与环境适配、状态与工件管理、验证反馈回路
- 关键问题：可靠性、可重复性、错误恢复、安全边界与长期任务控制
- 与测试、CI/CD、可观测性、安全合规的衔接

### 6.5 生产治理方法（扩展主题）

- 发布审批与人工接管
- 风险分级与回滚机制
- 跨团队协作规范
